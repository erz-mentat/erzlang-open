## CLI replay lane

Checked-in fixture packs can now be replayed directly from the CLI. The tracked `program-pack-index.json` intentionally uses relative child-pack entries and a declared replay order of `ingest-normalize`, `alert-routing`, then `dedup-cluster`, so index-target summaries and strict `--expected-selected-pack` contracts exercise the raw index-path lane instead of falling back to directory sort order:

```bash
erz pack-replay examples/program-packs/ingest-normalize --summary
erz pack-replay examples/program-packs/dedup-cluster --summary
erz pack-replay examples/program-packs/program-pack-index.json --summary
erz pack-replay examples/program-packs --include-pack "*cluster" --include-pack "*normalize" --summary
erz pack-replay examples/program-packs/program-pack-index.json --include-pack "*alert-routing" --exclude-pack "*dedup*" --summary
erz pack-replay examples/program-packs/dedup-cluster --fixture new_ops_time_miss --summary
erz pack-replay examples/program-packs/dedup-cluster --include-fixture "new_*" --exclude-fixture "*security*" --summary
erz pack-replay examples/program-packs/dedup-cluster --fixture-class runtime_error --summary
erz pack-replay examples/program-packs/dedup-cluster --summary-file /tmp/dedup.summary.txt
erz pack-replay examples/program-packs/dedup-cluster --summary --json-file /tmp/dedup.summary.json
erz pack-replay examples/program-packs/dedup-cluster --fixture-class runtime_error --fixture-class-summary-file /tmp/dedup-runtime-error.summary.txt
erz pack-replay examples/program-packs/dedup-cluster --summary --strict-profile clean
erz pack-replay examples/program-packs/dedup-cluster --summary --strict-profile dedup-cluster-clean
erz pack-replay examples/program-packs/ingest-normalize --summary --strict-profile ingest-normalize-clean

erz pack-replay examples/program-packs --summary --strict --expected-pack-count 4 --expected-total-pack-count 4
erz pack-replay examples/program-packs --strict --expected-fixture-class-counts ok=13,expectation_mismatch=0,runtime_error=0
```

The repo also carries refreshable replay-contract snapshots directly next to the packs: one `*.replay.expected.*` triple for each checked-in pack (`ingest-normalize`, `dedup-cluster`, `alert-routing`, `refs-handoff`) plus the aggregate `program-pack-index.replay.expected.*` set. Run `python3 scripts/refresh_program_pack_replay_contracts.py` to regenerate every per-pack summary/full-JSON/handoff-bundle sidecar and the aggregate index sidecars in one deterministic pass. Those checked-in JSON fixtures normalize internal pack paths to fixture-root-relative strings, so temp-copy refreshes stay byte-stable instead of leaking machine-local absolute roots.

Current scope: both fixture-matrix baselines (`baseline.json` or `*.baseline.json` with `rules` + `fixtures`) and inline statement baselines that carry embedded `ev` samples. Fixture-matrix packs can now pin `expected_action_plan` directly inside checked-in fixtures, so replay validates both the raw runtime `actions` and the materialized operator-facing plan in one lane. The new `refs-handoff/` pack extends that contract by pinning explicit `expected_resolved_refs` per fixture as well, giving operators a plain JSON surface for editing both the plan and the exact ref bundle that must materialize from the checked-in program. For inline baselines, `pack-replay` derives deterministic replay fixtures from the checked-in event statements plus the baseline rule set, so packs like `ingest-normalize/` no longer need a separate replay matrix just to reach the CLI lane, and replay JSON now includes `action_plan` plus `resolved_refs` whenever refs were materialized. For CI gates, `--fixture <id>` can be repeated for exact cases, while `--include-fixture <glob>` and `--exclude-fixture <glob>` build deterministic slices in canonical pack order, include/exakt first, exclude second; on multi-pack targets, `--include-pack <glob>` and `--exclude-pack <glob>` do the same one level higher for visible pack paths, meaning child directory names for collection dirs and the raw display paths from a pack index, so operators can carve out stable pack subsets without maintaining an extra pack-index file. `--fixture-class <ok|expectation_mismatch|runtime_error>` filters the replayed slice afterward by actual outcome class; `--summary` then reports `fixtures=<selected>/<total>` plus `fixture_classes=ok:<n>,expectation_mismatch:<n>,runtime_error:<n>`, while JSON consumers can read the same exclusive class labels per fixture under `fixtures[].fixture_class`, the aggregated counters from `summary.fixture_class_counts`, and the deterministic per-class fixture ids from `summary.fixture_class_ids`. Aggregate JSON envelopes also carry `selected_pack_paths`, `summary.total_pack_count`, and, when used, `include_pack_globs` plus `exclude_pack_globs`, so both the chosen pack slice and the pre-filter collection size remain machine-readable. `--summary-file <path>` now writes that same compact summary line for every pack run to a sidecar file, `--json-file <path>` writes the full JSON envelope side-by-side so summary stdout can hand off machine-readable results in the same run, `--handoff-bundle-file <path>` wraps the summary line, exit code, and full replay envelope into a single deterministic handoff JSON without changing stdout, `--fixture-class-summary-file <path>` remains available as the narrower compatibility path for fixture-class filtered runs, and `--output <path>` still mirrors the actual stdout lane. For the common checked-in green lane, `--strict-profile clean` is the generic short path: it auto-enables strict replay and gates `rule_source_status=ok` plus zero mismatches, zero expectation mismatches, and zero runtime errors without forcing brittle repeated fixture-id bundles into CI scripts. For the four checked-in packs, `--strict-profile ingest-normalize-clean`, `dedup-cluster-clean`, `alert-routing-clean`, and `refs-handoff-clean` go one step tighter by also pinning the expected `pack_id`, `baseline_shape`, total fixture count, and full `fixture_class_counts` histogram, so each pack can collapse to a single stable preset without extra count flags. For the checked-in full collection, `--strict-profile program-pack-index-clean` does the same at aggregate scope by deriving `expected_pack_count`, `expected_total_pack_count`, `expected_selected_pack_paths`, `expected_rule_source_status_counts`, `expected_fixture_class_counts`, `expected_action_plan_count`, `expected_resolved_refs_count`, and zeroed `expected_mismatch_field_counts` from the checked-in pack index plus the per-pack replay contracts. For expected non-green baselines, `--strict` or `--strict-profile <preset>` plus `--expected-pack-id <pack-id>`, `--expected-baseline-shape <fixture-matrix|inline-statements>`, `--expected-fixture-count`, `--expected-total-fixture-count`, repeatable `--expected-selected-fixture <id>`, repeatable `--expected-ok-fixture <id>`, repeatable `--expected-expectation-mismatch-fixture <id>`, repeatable `--expected-runtime-error-fixture <id>`, `--expected-fixture-class-counts ok=<n>,expectation_mismatch=<n>,runtime_error=<n>`, `--expected-mismatch-count`, `--expected-expectation-mismatch-count`, `--expected-runtime-error-count` and `--expected-rule-source-status` turns the replay into an exact contract, keeping the raw replay outcome under `replay_status` while `status` becomes the gate result. `--expected-selected-fixture` checks the final deterministic `selected_fixture_ids` list after exact/glob/class filtering in canonical pack order, and the class-specific fixture selectors check the exact deterministic ids under `summary.fixture_class_ids.ok`, `summary.fixture_class_ids.expectation_mismatch` and `summary.fixture_class_ids.runtime_error`, so selector drift, green-partition drift, and failure-partition drift still trip strict replay even when the aggregate counters stay unchanged. If exact ids are overkill, `--expected-fixture-class-counts` can instead hard-gate the full deterministic histogram under `summary.fixture_class_counts`. `--expected-pack-id` gives custom packs the same explicit identity gate that the checked-in `*-clean` presets already use internally, without forcing another named preset, and `--expected-baseline-shape` does the same for fixture-matrix versus inline-statement baseline form. `--expected-mismatch-count` tracks the broad `summary.mismatch_count` including runtime errors, while `--expected-expectation-mismatch-count` isolates pure expectation drift. On multi-pack targets, strict replay stays intentionally narrow: `--expected-pack-count`, `--expected-total-pack-count`, repeatable `--expected-selected-pack`, `--expected-rule-source-status-counts`, and aggregate `--expected-fixture-class-counts` are allowed, so collection replay can hard-gate both deterministic pack selection/order, the pre-filter collection size, and the visible cross-pack class histogram without pretending single-pack fixture selectors belong on an aggregate lane. That pairs directly with `--include-pack` and `--exclude-pack`, so selector drift can be gated without inventing a second aggregate-only policy layer, while `--expected-total-pack-count` still catches pack growth or shrink outside the selected subset. When that gate trips, the aggregate envelope keeps raw replay health under `replay_status`, flips `status` to the strict result, and reports the exact `pack_count`, `total_pack_count`, `selected_pack_paths`, `rule_source_status_counts`, or `fixture_class_counts` drift under `strict_profile_mismatches`.
